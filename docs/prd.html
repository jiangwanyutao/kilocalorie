01产品概述

是什么，叫什么，怎么装。

产品名称
千卡日记（英文 Kaily · 备选 KcalDiary）
形态
H5 PWA（HTTPS）· 加到 iOS/Android 桌面即"轻应用"，不上架应用商店
Slogan 候选
"吃点什么，记一笔" / "每一口，都值得被认真看见" / "把今天的胃口，留个数字"（三选一，未定）

iOS 15+ Safari · Android 10+ Chrome · 移动优先，桌面端可用不重点优化

V1 全免费，数据模型预留付费墙字段（AI 拍照识别、AI 搭子对话未来上会员）

02目标用户与价值

给谁做，给他们解决什么。

用户画像

  • 18–45 岁，在意饮食但对薄荷健康 / MyFitnessPal 觉得"太重、太贵、太麻烦"
  • 需求分布：减重（约 55%）· 健身增肌（约 25%）· 单纯规律记录（约 20%）
  • 使用场景：一餐结束后 30 秒内快速记录；每天早晨或睡前看昨日总结

核心价值主张

  1. 三秒记一餐 —— 拍照识别 / 常用食物一键复用 / 内置库快速搜 / 外卖订单截图解析
  2. 一屏看得懂 —— 今日卡路里缺口、宏营养分布、饮水、体重同屏
  3. 陪你坚持 —— AI 搭子对话陪伴，可选人设，从毒舌教练到温柔知心
  4. 数据是你的 —— 可导出 JSON，无广告，不做用户画像转卖

与市面同类的差异化

  • PWA 加桌面而非应用商店 —— 无审核、无更新等待、无预留量
  • 粒度可选 —— 严格时切克，日常时用份，同一 App 满足两种心情
  • AI 拍照 + AI 搭子 下沉到 V1 免费额度 —— 主流工具多是付费墙功能

03关键决策记录

本次讨论沉淀下来的定案，后续所有工作都以此为准。

技术决策
  • 前端：Vue 3 + Vite + Pinia + vue-router + vite-plugin-pwa + Dexie（IndexedDB 离线缓存）
  • 后端：Node.js（Fastify）+ Prisma ORM + PostgreSQL 15+ + Redis（会话/限流）
  • 部署：Docker Compose 单机部署到用户内网 Linux 服务器
  • 域名：用户提供已备案的 HTTPS 域名（iOS PWA 硬性要求）
  • AI 视觉：DeepSeek V4 Vision（若通道不稳，切阿里 qwen-vl-plus）· 通过 VisionProvider 适配器接入
  • AI 对话：DeepSeek Chat（推理模型）+ 自建饮食营养 RAG 知识库
  • 图片存储：MVP 用本地 Docker 卷 /uploads，量大再迁 OSS
  • 邮件推送：阿里云邮件推送（用户注册验证 + 密码找回）
产品决策
  • 用户体系：完整注册（邮箱 + 密码 + 邮箱验证）· 无手机号 · 无第三方登录
  • 粒度：份 / 克双档，用户按需切换，默认份量
  • 目标人群：通用饮食管理，不偏减脂或增肌某一场景
  • 付费策略：V1 全免费，数据库预留 subscription 字段与 ai_quota 记次数字段
  • Apple Health：PWA 无法直接读 HealthKit；退路是 XML 一次性导入 + iOS Shortcuts 自动化模板
合规决策
  • 服务器在大陆，域名已备案 → 可公开推广
  • 必备页面：用户协议、隐私政策、Cookie 说明、备案信息
  • 用户可主动删除账号，历史记录 30 天内彻底物理删除
  • AI 对话内容 + 拍照原图默认加密存储 30 天，用户可清除
明确不做（V1）
  • iOS Widget / 灵动岛 / Apple Watch / ARKit 分量测距（需原生 App，PWA 无解）
  • Apple HealthKit 双向同步（同上）
  • 条码扫描（iOS Safari 不支持 BarcodeDetector）
  • 智能体脂秤蓝牙同步（iOS Safari 不支持 Web Bluetooth）
  • 社交、评论、动态、公开 UGC 食物库
  • 微信 / QQ / Apple 第三方登录
  • 饮食推荐算法 / 全套食谱推荐（作为 V1.5+ 内容运营需求）

05核心用户流程

五条走完就能上路的路径。

流程 A · 首次上手

  1. 着陆页 → 注册
    填邮箱 + 密码 → 收激活邮件 → 点激活链接
  2. 身体信息
    性别 / 出生年 / 身高 / 体重 / 活动等级（5 档滑块）
  3. 目标推荐
    系统按 TDEE 推荐每日卡路里、宏比、饮水；用户可覆盖
  4. 提醒设置
    默认早 08:00 / 午 12:00 / 晚 18:30，可跳过
  5. 进入首页
    看到今日 0 kcal / 目标 X kcal，出现"记第一餐"引导

流程 B · 日常记一餐

  1. 点底部记录按钮
    选餐次（默认按时间推断早/午/晚/加餐）
  2. 选食物
    四个来源：常用 / 搜内置库 / 私人库 / 拍照识别 / 外卖截图
  3. 确认份量
    默认份，可切克；实时看卡路里 + 宏营养
  4. 保存
    回到首页，今日环 + 缺口立即更新

流程 C · 拍照识别

  1. 拍照或从相册选
    前端压缩至 1024 px 短边
  2. 上传后端 → VisionProvider
    默认 DeepSeek V4 Vision，返回结构化候选列表 + 置信度
  3. 用户确认候选
    每个候选带估算份量，用户可增删改
  4. 保存为一餐
    原图挂在该餐次，可回看

流程 D · Apple Health 数据接入

  1. 路径 1：一次性历史导入
    iOS 系统健康 App → 头像 → 导出所有健康数据 → 用户在千卡日记选 ZIP 上传 → 后端解析 export.xml
  2. 路径 2：Shortcuts 自动化
    千卡日记提供 iCloud 分享的 Shortcut 模板，用户装到 iOS 快捷指令 App，配 Automation 每日 8:00 自动读体重 / 步数 POST 到 API
  3. 后端处理
    解析后写入 WeightEntry / StepsEntry，标记来源 source=apple_health

流程 E · AI 搭子对话

  1. 选人设
    4 种预置：温柔知心 / 毒舌教练 / 闺蜜风 / 专业营养师；可自定义头像与聊天背景
  2. 提问
    用户输入问题（如"这个鸡腿汉堡能吃吗"）
  3. 后端 RAG
    先查私域营养库找相关条目 → 拼进 prompt → DeepSeek 出答复
  4. 展示 + 附免责
    每条对话底部固定文案："AI 内容仅供参考，不构成医疗建议"

07关键业务规则

推荐值背后的公式。所有数值用户可覆盖。

BMR · 基础代谢（Mifflin-St Jeor）

男：BMR = 10 × 体重kg + 6.25 × 身高cm − 5 × 年龄 + 5
女：BMR = 10 × 体重kg + 6.25 × 身高cm − 5 × 年龄 − 161

TDEE · 每日总消耗

活动等级说明系数
静坐坐办公室 / 极少运动1.20
轻度每周 1–3 次轻度运动1.375
中度每周 3–5 次中强度运动1.55
高度每周 6–7 次高强度运动1.725
极重体力劳动 / 每日两次训练1.90

卡路里目标推荐

  • 维持：TDEE
  • 温和减脂：TDEE − 300
  • 标准减脂：TDEE − 500（不低于 BMR × 1.1）
  • 温和增肌：TDEE + 250

宏营养素默认比例

碳水 50% · 蛋白 25% · 脂肪 25%（1 g 碳水 / 蛋白 = 4 kcal，1 g 脂肪 = 9 kcal）

饮水推荐

日饮水 ml = 体重 kg × 30（上限 3500 ml）

提醒时段默认

早餐 08:00 · 午餐 12:00 · 晚餐 18:30 · 静默 22:00 – 07:00

饮品类型折算（饮水记录）

白水 = 1.00 · 茶 = 0.95 · 咖啡（<300 ml/日） = 0.90
果汁 = 0.85（含糖负担另计） · 汤 = 0.80 · 奶茶 = 不计入水

AI 用量策略（V1 免费，为付费墙留数）

  • 拍照识别：免费用户每日 5 次，超出提示"未来会员可无限次"
  • AI 搭子对话：免费用户每日 20 条，超出同上
  • 外卖截图解析：每日 3 次
  • 用量记 AiUsageQuota，付费墙上线时改成"会员无限，普通用户额度不变"

08平台约束与退路

PWA 相对原生 App 的硬约束，以及我们对每一项的应对。

受限能力PWA 可行度退路 / 备注
Apple HealthKit 读写完全不能XML 一次性导入 + iOS Shortcuts 自动化模板（体重 / 步数每日同步）
iOS Widget不能V1 不提供，长期看考虑轻量 iOS 桥接 App
灵动岛 / Live Activities不能V1 不提供
Apple Watch 版不能V1 不提供
ARKit 分量测距不能用份量估算兜底，AI 拍照识别提示"建议俯拍完整餐具"
条码扫描iOS Safari 不支持 BarcodeDetectorAndroid Chrome 可用（能力检测降级）；V1 不做
Web Bluetooth（蓝牙体脂秤）iOS Safari 不支持手动录入体脂率
本地通知受限但可用需 iOS 16.4+ 且加桌面后；早期系统用户拿不到
语音识别可用但差Web Speech API 精度弱，V1 作为 P2 尝试
决策解释：这些"不能做"是 iOS/浏览器平台约束，不是我们能力不足。若未来 iOS 用户对这些能力呼声高，可评估追加一个极小的 iOS 原生桥接 App（V2+）。V1 用纯 PWA 路径换取快速上线、免上架、跨端可用。

09非功能需求

性能、隐私、可用、可迁移。

性能
首屏 LCP < 2.5s（4G）；记录一餐提交 < 500ms（P95）；AI 拍照识别 < 5s；离线可查看已缓存的记录
可用性
iOS 15+ Safari / Android 10+ Chrome / 桌面 Safari + Chrome 现代版本
PWA 完整度
Lighthouse PWA 得分 ≥ 90；iOS 加桌面全屏无地址栏；splash / favicon / apple-touch-icon 齐
隐私
密码 bcrypt(12)；照片私有；不做广告；不共享数据给第三方；用户可清除全部数据
合规
合规页齐全；域名底部展示备案号；账号删除请求 T+30 天内物理删除；AI 内容附免责声明
可迁移
用户可导出完整 JSON（含饮食、体重、目标、私人食物库、AI 对话）
备份
每日 pg_dump 到宿主机 /backup，保留最近 14 天
监控
后端 pino 结构化日志；异常报警可选（V1 内不做，留 hook）
限流
Redis 令牌桶：注册接口 3/min/IP、登录 10/min/IP、AI 接口 按用户 quota

11风险与待议

已知会踩的坑，以及需要你/我在开工前定的事。

已知风险
  • DeepSeek V4 Vision API 官方文档尚未列图片接口。W6 如接不上，切 qwen-vl-plus（阿里通义）；VisionProvider 适配器已计划
  • 内置食物库精度：100 条中式食物营养值需人工核对，误差过大会误导用户
  • 邮件推送：预计每日 < 500 条，阿里云邮件推送月费 < ¥100，注意 SPF/DKIM 配置避免进垃圾箱
  • PWA 通知在 iOS：仅 iOS 16.4+ 且加桌面后才支持，早期 iOS 用户拿不到提醒
  • 图片存储：本地卷方案在用户量 > 1000 后需评估迁 OSS
  • 外卖截图解析：美团 / 饿了么 UI 会改版，OCR 提示词需持续维护
  • Shortcut 模板：iOS 版本升级可能破坏兼容，需在支持文档里维护多版本
